# Робота з GitHub у PlatformIO (VS Code)

Ця інструкція описує базовий робочий цикл для збереження змін, керування версіями та синхронізації коду проекту **Emulator_OBD_II**.

---

## 1. Підготовка (.gitignore)
Перш ніж надсилати код, переконайтеся, що тимчасові файли не потрапляють у репозиторій. Файл `.gitignore` у корені проекту має містити:
```text
.pio/
.vscode/
*.log
.DS_Store
```
*Це дозволить тримати репозиторій чистим, зберігаючи лише вихідний код та конфігурацію.*

---

## 2. Збереження змін (Commit & Push)
Виконуйте ці кроки щоразу, коли реалізували нову функцію або виправили помилку:

1.  **Stage (Індексація):**
    *   Відкрийте вкладку **Source Control** (Ctrl+Shift+G).
    *   Натисніть **+** (Stage All Changes) або біля конкретних файлів.
2.  **Commit (Фіксація):**
    *   У полі повідомлення напишіть опис змін (наприклад: `Додано симуляцію пропусків запалювання P0300`).
    *   Натисніть кнопку **Commit**.
3.  **Push (Відправка):**
    *   Натисніть **Sync Changes** або `...` -> **Push**, щоб відправити зміни на сервер GitHub.

---

## 3. Керування версіями та гілками

### Збереження поточного стану в окрему гілку
Якщо ви хочете зберегти "зліпок" проекту (наприклад, версію без дисплея) як окрему гілку:
1. Створіть нову гілку: `git checkout -b назва-гілки`
2. Додайте зміни: `git add .`
3. Зробіть комміт: `git commit -m "Опис стану"`
4. Відправте на GitHub: `git push -u origin назва-гілки`

### Робота з функціональними гілками
Для експериментів (наприклад, нова логіка UDS):
1.  Натисніть на `main` у нижньому лівому куті.
2.  Оберіть **Create new branch...** -> назва `feature-uds-logic`.
3.  Після завершення та перевірки, злийте її з `main` через Pull Request на сайті GitHub.

---

## 4. Оновлення та завантаження коду

*   **Pull:** Якщо ви працюєте на декількох ПК, перед початком роботи натисніть `...` -> **Pull**, щоб отримати останні зміни з хмари.
*   **Clone:** Щоб розгорнути проект на новому ПК, скористайтеся командою `Git: Clone` у палітрі команд (F1) і вставте посилання на репозиторій. PlatformIO автоматично завантажить необхідні бібліотеки при першому збиранні.

---

## Поради щодо якості коду

1.  **Атомарні комміти:** Один комміт = одна зміна. Не змішуйте виправлення інтерфейсу TFT та логіку CAN-шини в одному збереженні.
2.  **Описи:** Пишіть повідомлення англійською або українською мовою, але будьте конкретними. Замість "Update", пишіть "Fix: correct DTC offset on TFT display".
3.  **Перевірка перед Push:** Завжди робіть `Build` проекту перед тим, як відправляти код на GitHub, щоб переконатися, що він компілюється.
